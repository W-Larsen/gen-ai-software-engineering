## 🧱 Architecture Decisions

- **In-memory store** (`src/services/transactionStore.js`): simple array storage, no DB.
- **Validation layer** (`src/validators/transactionValidator.js`): centralized request and filter validation.
- **Routing layer** (`src/routes/transactions.js`): endpoint handling, filtering, and balance logic.
- **App bootstrap** (`src/app.js`, `src/index.js`): express configuration and server startup.

Server-managed transaction fields:

- `id`: auto-generated by server (`crypto.randomUUID()`)
- `timestamp`: auto-generated ISO 8601 datetime
- `status`: always set to `"completed"` on creation

## 🔌 API Endpoints

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/transactions` | Create a new transaction |
| `GET` | `/transactions` | List transactions (supports filters) |
| `GET` | `/transactions/:id` | Get transaction by ID |
| `GET` | `/accounts/:accountId/balance` | Get account balance |
| `GET` | `/accounts/:accountId/summary` | Get account transaction summary |
| `GET` | `/accounts/:accountId/interest?rate=0.05&days=30` | Calculate simple interest from current balance |

### Supported filters for `GET /transactions`

- `accountId=ACC-12345`
- `type=deposit|withdrawal|transfer`
- `from=YYYY-MM-DD` or ISO datetime
- `to=YYYY-MM-DD` or ISO datetime

Filters can be combined (AND semantics).

### `GET /accounts/:accountId/summary` response fields

- `totalDeposits`: account inflows (`deposit` + incoming `transfer`)
- `totalWithdrawals`: account outflows (`withdrawal` + outgoing `transfer`)
- `transactionCount`: number of related transactions
- `mostRecentTransactionDate`: latest related timestamp or `null`

### `GET /accounts/:accountId/interest` rules

- Formula: `interest = balance * rate * (days / 365)`
- `rate` must be numeric and `>= 0`
- `days` must be an integer and `>= 0`
- Current balance can be negative, resulting in negative interest
- Empty accounts return zero balance and zero interest

---

## 🧪 Validation Rules

- `amount` must be positive and have at most 2 decimal places
- `fromAccount` and `toAccount` must match `ACC-XXXXX` (`X` is alphanumeric)
- `currency` must be a valid ISO 4217 code (full code set)
- `type` must be one of `deposit`, `withdrawal`, `transfer`
- Client cannot provide `id`, `timestamp`, `status` in create requests

Example validation response:

```json
{
  "error": "Validation failed",
  "details": [
    { "field": "amount", "message": "Amount must be a positive number" }
  ]
}
```

---

## ⚖️ Balance Calculation Rules

- `deposit`: adds amount when `toAccount` matches
- `withdrawal`: subtracts amount when `fromAccount` matches
- `transfer`: subtracts from sender and adds to receiver

---

## 🚦 Rate Limiting

- Maximum `100` requests per minute per IP
- Applied globally to all endpoints
- Exceeded limit response: HTTP `429`

```json
{
  "error": "Too many requests"
}
```
