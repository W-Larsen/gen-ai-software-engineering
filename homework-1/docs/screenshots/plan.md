## Simple Banking Transactions API (Tasks 1-3) — Implementation Plan

### Summary
Build a Node.js + Express REST API with in-memory transaction storage, full validation, and filtered history queries, plus automated tests for each of the first 3 tasks.  
API will expose:

- `POST /transactions`
- `GET /transactions`
- `GET /transactions/:id`
- `GET /accounts/:accountId/balance`

Server-managed fields (`id`, `timestamp`, `status`) are not accepted from clients; they are generated by the server, with `status = "completed"` on creation.

### Implementation Changes
1. Project setup
- Initialize Node project with `express` for API, `supertest` + test runner (`jest`) for integration tests.
- Add scripts: `start`, `dev`, `test`.
- Add `.gitignore` for `node_modules`, coverage artifacts, and env files.

2. API structure and core behavior (Task 1)
- Create app entry and route/controller split (keep minimal, no DB layer).
- In-memory store: array of transaction objects (lives for process lifetime only).
- `POST /transactions`:
  - Accept business fields: `fromAccount`, `toAccount`, `amount`, `currency`, `type`.
  - Generate `id` (UUID), `timestamp` (ISO 8601), `status` (`completed`).
  - Return `201` with created transaction.
- `GET /transactions`: return all transactions (`200`) before filters are applied.
- `GET /transactions/:id`: return matching transaction or `404`.
- `GET /accounts/:accountId/balance`:
  - Validate `accountId`.
  - Compute using standard ledger rules:
    - `deposit`: `+amount` if `toAccount === accountId`
    - `withdrawal`: `-amount` if `fromAccount === accountId`
    - `transfer`: `-amount` for sender, `+amount` for receiver
  - Return `200` with `{ accountId, balance }`.
- Basic error middleware for consistent JSON errors and status mapping.

3. Validation layer (Task 2)
- Central transaction validator used by `POST /transactions`.
- Enforce:
  - `amount` positive and max 2 decimal places.
  - `fromAccount` and `toAccount` match `^ACC-[A-Za-z0-9]{5}$`.
  - `currency` exists in a full ISO 4217 code set (uppercase normalized).
  - `type` in `deposit | withdrawal | transfer`.
- Validation failure response (`400`):
  - `{ "error": "Validation failed", "details": [{ "field": "...", "message": "..." }] }`
- Reject client-supplied `id`, `timestamp`, or `status` with validation errors.

4. Transaction history filtering (Task 3)
- Extend `GET /transactions` query handling:
  - `accountId`: include transactions where account is either sender or receiver.
  - `type`: exact match for valid type.
  - `from`, `to`: filter by `timestamp` date range (inclusive).
- Combined filters use AND semantics.
- Invalid filter inputs return `400` with meaningful details.

### Test Plan
1. Task 1 tests (core API)
- Creates transaction successfully (`201`) with auto-generated fields.
- Lists transactions (`200`) and includes created items.
- Fetches transaction by valid id (`200`) and unknown id (`404`).
- Computes balance correctly for deposit/withdrawal/transfer mix.

2. Task 2 tests (validation)
- Rejects zero/negative amount and >2 decimal places (`400` + details).
- Rejects malformed accounts (`400` + field-level messages).
- Rejects invalid currency code (`400`).
- Rejects invalid type (`400`).
- Rejects requests containing `id/timestamp/status` (`400`).

3. Task 3 tests (history filters)
- Filters by `accountId`.
- Filters by `type`.
- Filters by date range.
- Applies multiple filters together correctly.
- Returns `400` for malformed `from/to` date or invalid query params.

### Assumptions and Defaults
- Runtime: Node.js with Express.
- Persistence: in-memory only; data resets on server restart.
- `status` is server-set to `"completed"` for new records.
- Currency validation uses a full ISO 4217 dataset (not a small hardcoded subset).
- Date range filtering is inclusive and based on parsed transaction timestamps.
